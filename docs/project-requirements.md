# Backend Requirements Specification

This document defines the **backend requirements** needed to fully support the current UI for the **Osimi Digital Library**, including **Dashboard**, **Ingestion**, and **Objects**. It is intended as a handoff document for backend engineers and as a contract between frontend and backend.

---

## 1. Core Concepts

### Identifiers

- **upload_id (batch_id)**
  - User-provided identifier
  - Used by the user/UI to track an ingestion
  - May be reused intentionally (backend must handle collisions explicitly)

- **object_id**
  - System-generated, immutable
  - Primary internal identifier for all ingested objects
  - Assigned at successful ingest completion
  - Format: `OBJ-YYYYMMDD-XXXXXX`
    - `YYYYMMDD` is UTC ingest date
    - `XXXXXX` is a random/ULID segment
  - Opaque and safe to expose as a stable reference
  - Distinct from `upload_id`/`batch_id` (user tracking ID)

### Object ID Generation

- Generated by the backend when an ingestion completes
- Guaranteed unique with no collisions
- Immutable once assigned
- Used as the primary key in the database
- Embedded in storage paths and metadata (e.g., `ingest.json`)

---

## 2. Top-Level Entities

### Ingestion
Represents a user-initiated ingestion request.

Responsibilities:
- Accept files
- Track progress and errors
- Spawn processing pipelines
- Produce one or more objects

### Object
Represents a durable, ingested item.

Responsibilities:
- Store metadata
- Reference original and derived artifacts
- Maintain provenance to ingestion

---

## 3. Authentication & Authorization

Backend must support:

- Session or JWT-based authentication
- Tenant scoping (even if single-tenant initially)
- Roles:
  - `viewer` – read-only
  - `operator` – create and manage ingestions
  - `admin` – destructive actions, user management

All endpoints must be tenant-aware.

### Session Persistence Requirements

- Session authentication must be persisted in the database (in-memory-only sessions are not sufficient)
- Session tokens must be stored as non-reversible hashes
- Sessions must support expiry and explicit revocation (logout)
- Session validation must enforce active user, active tenant, and active tenant membership
- Authentication lifecycle events (login/logout/session rejection) must be persisted for audit

## 4. Ingestion Lifecycle

### Supported States

```
DRAFT → UPLOADING → QUEUED → PROCESSING → COMPLETED
                             ↘ FAILED
                             ↘ CANCELED
```

### Backend Responsibilities

- Create ingestion record
- Accept uploaded files
- Track per-file and per-ingestion status
- Start async processing
- Support retry and cancel
- Emit events for UI updates

### Behavioral Requirements

- Enforce authorization per action and state
- Ensure idempotency for create/submit/retry operations
- Validate inputs and normalize error responses
- Apply concurrency limits per tenant/user
- Persist audit events alongside emission for UI updates
- As part of the v1 long polling to be used as a events delivery strategy (later expandable to SSE or WebSocket) 

### Worker Lease Protocol

To support outbound-only workers and avoid race conditions, ingestion processing must use a lease.

- A lease is granted via `POST /api/ingestions/lease` and is exclusive per ingestion
- Lease grant is atomic (select + update in a single transaction) to ensure only one worker receives a batch
- Lease has a TTL of 5 minutes and must be renewed via heartbeat
- If a lease expires before completion, the ingestion automatically returns to `QUEUED`
- A periodic redundancy sweep must re-queue any `PROCESSING` ingestion with an expired lease
- Lease ownership is enforced using a signed `lease_token` issued by the VPS at grant time
- `worker_id` is optional metadata provided by the worker; it may be dynamic per process run

### Lease Model

- Lease state is stored in `ingestion_leases`
- At most one active lease per ingestion (`released_at` is null and `lease_expires_at` > now)
- Lease ownership is validated by signature + matching lease row
- Leases are created on grant and marked released on completion/cancel

### Staging Cleanup & Retention

- `COMPLETED`: retain staging files for 7 days, then purge
- `FAILED` and `CANCELED`: retain staging files for 14 days, then purge
- `DRAFT`, `UPLOADING`, `QUEUED`, `PROCESSING`: retain indefinitely

#### Stuck Attention

- If an ingestion remains in `UPLOADING` or `PROCESSING` with no progress/heartbeat for a configurable threshold, it must be flagged for user attention (no automatic purge)

### Integrity Checks

- Upload commit must include a SHA-256 checksum for each file
- Backend verifies checksum on commit before accepting the file
- Worker re-verifies checksum after downloading from staging
- Mismatches must emit `FILE_FAILED` and may fail the ingestion or trigger retry

## 5. Object Management

Backend must support:

- Listing objects with filters (type, date, tags)
- Object detail view
- Linking objects to source ingestion
- Serving original and derived artifacts
- Updating curated metadata (human-in-the-loop)

### Ingest Manifest Artifact

- The worker must upload `ingest.json` on successful completion
- `ingest.json` is stored as an object artifact (`kind = ingest_json`)
- Storage key follows object artifact layout (e.g., `tenants/{tenant}/objects/{object_id}/artifacts/ingest.json`)
- `ingest.json` is retained permanently as part of the object record
- VPS may use `ingest.json` to reconcile or validate event history

Backend must provide:

- Aggregated statistics
  - total ingestions
  - total objects
  - processed today / week
  - failed count

  - ingestion events

### Event Ingestion (Worker → VPS)

Purpose: allow the private worker to report ingestion progress and outcomes to the VPS for UI updates and audit.

#### Event Schema (minimum)

- `event_id` (UUID)
- `event_type` (enum)
- `ingestion_id` (UUID)
- `object_id` (string, nullable)
- `timestamp` (UTC)
- `payload` (JSON)

#### Delivery Semantics

- At-least-once delivery; VPS must deduplicate by `event_id`
- Ordering is best-effort and not guaranteed
- VPS must reject events without a valid lease

#### Storage

- Events are persisted in `object_events`
- Ingestion-level events have `object_id = null`

#### Recommended Event Types

- `INGESTION_SUBMITTED`, `INGESTION_QUEUED`, `INGESTION_PROCESSING`
- `INGESTION_COMPLETED`, `INGESTION_FAILED`, `INGESTION_CANCELED`
- `LEASE_GRANTED`, `LEASE_RENEWED`, `LEASE_EXPIRED`, `LEASE_RELEASED`
- `FILE_VALIDATED`, `FILE_FAILED`
- `PIPELINE_STEP_STARTED`, `PIPELINE_STEP_COMPLETED`, `PIPELINE_STEP_FAILED`
- `OBJECT_CREATED`, `ARTIFACT_CREATED`

## 9. Async Processing Contract

Backend must define pipeline steps with state tracking:

- VALIDATE
- NORMALIZE
- DERIVE_PREVIEWS
- OCR
- TRANSCRIBE
- EXTRACT_METADATA
- INDEX

Each step must emit events and record failures.

## 10. Storage Abstraction

Backend must expose a storage interface supporting:

- put / get / head / list
- signed URLs
- range requests (audio/video)

### Signed URL Policy

- Upload URLs are issued per file and are valid for 60 minutes
- Upload URLs are `PUT` only and scoped to a specific staging key
- Upload requests must include `Content-Type` and `Content-Length`
- Upload commit requires a SHA-256 checksum; backend verifies integrity
- Upload URLs are reusable for the same file/key until expiry
- Re-presign is allowed for the same file if a URL expires mid-upload
- Worker download URLs are issued only with a valid lease
- Worker download URLs are `GET` only, valid for 5 minutes, and refreshable on heartbeat
- URL issuance must be audited (who, what, when, TTL)

### Recommended Layout

```
tenants/{tenant}/ingestions/{ingestion_id}/original/
tenants/{tenant}/objects/{object_id}/artifacts/
```

## 11. Non-Functional Requirements

- Cursor-based pagination
- Idempotent ingestion + upload operations
- Structured logging with request_id
- Metrics for throughput and failures
- Audit trail via object_events
- Rate limiting for uploads

### Testing & Verification

- Lease exclusivity: only one worker can lease a batch at a time (race-safe)
- Lease expiry: expired leases re-queue correctly and are picked up by another worker
- Signed URL constraints: method, TTL, content-type, and content-length are enforced
- Integrity checks: SHA-256 validation on upload commit and worker re-download
- Event ingestion: idempotent `event_id` handling and tolerance to out-of-order delivery
- Staging retention: purge rules per state and stuck-attention alerts are triggered
- Auth/tenant scoping: all endpoints enforce tenant isolation and role permissions
- Auth persistence: sessions are durable, expirable, revocable, and token hashes (not raw tokens) are stored
- Auth audit: login/logout/session rejection events are persisted and queryable by request ID
